# TaeKwonTrack — Full Codebase Sweep

## PRIORITY 1 — Attendance Fixes (Do These Now)

### 1. Calendar off-by-one in month navigation
**File:** `src/components/AccountDashboards/AdminFeatures/AttendanceRecords/AttendanceCalendar.tsx`

```ts
// BROKEN
const newDate = new Date(
  date.getFullYear(),
  date.getMonth() + 1 + (direction === "next" ? 1 : -1),
  1
);

// FIXED
const newDate = new Date(
  date.getFullYear(),
  date.getMonth() + (direction === "next" ? 1 : -1),
  1
);
```
`getMonth()` is already 0-indexed. The extra `+1` means "Previous" stays on current month and "Next" skips a month.

### 2. isLoading stuck forever when school has 0 students
**File:** `src/context/AttendanceContext.tsx`

```ts
// BROKEN
if (students.length > 0) {
  fetchExistingAttendance();
}

// FIXED
if (students.length > 0) {
  fetchExistingAttendance();
} else {
  setIsLoading(false);

```
If a school hasn't added students yet, the loading spinner never clears and the attendance page is unusable.

### 3. upsert has no conflict target — can cause duplicate inserts
**File:** `src/api/Attendance/attendanceRequests.ts`

```ts
// BROKEN
const { data, error } = await supabase.from("attendance_records").upsert(records);

// FIXED
const { data, error } = await supabase
  .from("attendance_records")
  .upsert(records, { onConflict: "student_id,date" });
```
Without specifying onConflict, Supabase may insert duplicates instead of updating. You have a unique index on (student_id, date) — use it.

The "Today's Attendance" stat card always shows 0 or silently errors because it queries a non-existent table.

### 6. AttendanceContext is a hook, not a context — no shared state
**File:** `src/context/AttendanceContext.tsx`

The file is named `AttendanceContext.tsx` but exports a plain custom hook with no Provider. Every component that calls `useAttendance()` gets its own isolated copy of state. This will break the dashboard stat card (which reads attendance count separately) and will cause issues when you add the parent portal. Should be converted to a proper context with a Provider before more features are added.

### 8. SchoolContext fetches metrics from a `sales` table that doesn't match your schema
**File:** `src/context/SchoolContext.tsx`

```ts
const { data: salesData } = await supabase
  .from("sales")
  .select("amount")
  .eq("school_id", schoolId);
```
Your `sales.sql` schema doesn't include a `school_id` column on the sales table (it references `student_id` instead). This query will always return 0 or error. The "Today's Money" stat card is broken.

### 9. loadStudents fetches ALL students then filters client-side
**File:** `src/context/SchoolContext.tsx`

```ts
// INEFFICIENT — fetches every student in the DB
const allUsers = await getStudents();
const filtered = allUsers.filter(
  (user) => user.role === "Student" && user.school_id === targetSchoolId
);

// getStudents() already supports schoolId — just use it
const filtered = await getStudents(targetSchoolId);
```
At 200 students this is fine, but as you scale to multiple schools this downloads every student record from every school before filtering. `getStudents()` in `studentRequests.ts` already accepts a `schoolId` param — just pass it.

## PRIORITY 3 — Architecture / Code Quality

### 15. Duplicate API files — AppUserRequests.ts vs UserService.ts
**Files:** `src/api/AppUserRequests/AppUserRequests.ts` AND `src/api/AppUserRequests/UserService.ts`

Two files in the same folder both export `createUser`, `updateUser`, `deleteUser` with different signatures. One uses `User`, one uses `UserProfile`. Neither appears to be actively used in the codebase. These should be consolidated or removed.

---

### 16. sales.sql has two CREATE TABLE statements
**File:** `src/sql/sales.sql`

```sql
-- First definition (simple, with user_id)
create table sales ( ... user_id uuid ... );

-- Second definition (detailed, with student_id, payment_type, etc.)
CREATE TABLE sales ( ... student_id BIGINT ... );
```
Running this migration will fail on the second `CREATE TABLE`. Pick one (the detailed second version matches your TypeScript types). Delete the first.

---

### 17. student_renewals.sql uses `id` but the app expects `renewal_id`
**File:** `src/sql/student_renewals.sql`

```sql
-- SQL uses 'id'
constraint student_renewals_pkey primary key (id)

-- But every API call and type uses 'renewal_id'
.eq("renewal_id", renewalId)
```
This mismatch was partially addressed before but the SQL file still shows the old schema. Make sure your migration ran the updated version with `renewal_id` as the primary key column name.

### 18. StatCards reads `schoolData.sales` and `schoolData.attendance` as numbers but they return 0

## PRIORITY 4 — Future / Pre-Launch

### 21. No loading or error state on SchoolManagement fetchSchool
The `fetchSchool` in `SchoolManagement.tsx` calls directly into Supabase with no error boundary. If the school isn't found (fresh signup), the component renders blank with no guidance.

### 22. Footer has placeholder contact info
`src/components/Footer.tsx` — phone, email, address are all placeholder strings. Fine for now but needs updating before any real users see it.

### 23. Announcements are entirely hardcoded
All announcement UI reads from `dummyInfo.ts`. Create/Edit/Delete call `console.log`. Not wired to Supabase at all.

### 24. Sales still uses mock data in production path
`salesApi.ts` simulates API calls with `setTimeout` and in-memory state. The Supabase sales table and schema exist but the API layer hasn't been wired. This is already in your todo list but worth flagging.

### 25. No 404 route defined
`AppRouter.tsx` has no catch-all `<Route path="*">`. Navigating to any undefined URL renders nothing with no feedback.

### 26. index.css loads Tailwind twice
Both `index.css` and `App.css` contain the full `@tailwind base/components/utilities` directives. This doubles the CSS output. Remove the three lines from `App.css` and keep them only in `index.css`.
